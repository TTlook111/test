# 企业级 NL2SQL 平台项目构想

## 1. 项目理解

我理解你的想法是：做一个面向企业内部员工的数据问答平台，系统直接连接公司内多个业务数据库。员工不需要懂 SQL，也不需要知道具体表名、字段名、库名，只需要通过自然语言提问，就能得到：

- 查询结果数据
- 汇总分析结果
- 图表或看板
- **跨库、跨产品的数据组合结果（核心亮点）**

这个项目的核心难点不只是“把自然语言翻译成 SQL”，而是要解决企业真实数据环境中的以下问题：

- 数据分散在多个数据库、多个产品系统中，形成数据孤岛
- 同名字段含义不同，异名字段含义相同，跨库联合时极易发生语义冲突
- 数据库 schema 只能说明结构，不能完整表达业务含义
- 很多字段没有注释，甚至技术人员也不完全清楚真实用途
- **一条查询结果可能依赖多张表、多个异构数据库，甚至依赖复杂的业务口径和数据血缘**

所以，这个项目本质上不是一个简单的 NL2SQL 工具，而是一个结合了：

- 语义建模
- 业务知识沉淀
- 数据治理
- 数据血缘
- 可信度管理
- 人机协同学习

的企业级“智能数据访问平台”。

## 2. 项目目标

这个平台的目标可以定义为：

> 让企业员工通过自然语言安全、准确、可解释地获取企业内部数据，并持续沉淀业务知识与数据治理能力。

具体目标包括：

- 降低数据使用门槛，让非技术人员也能查数据
- 降低对 BI、数据分析师、开发人员的重复性咨询依赖
- 打通多数据库、多系统的数据访问路径
- 让 AI 不只是依赖 schema，而是依赖“结构 + 业务知识 + 治理信息”
- 让系统在使用过程中不断学习，提高字段理解与查询准确率

## 3. 你这个想法的核心创新点

## 3.1 不只看 schema，还引入业务知识文档

你提到的 `skills.md` 很关键。它可以理解为 AI 的“业务语义说明书”，用于补足数据库 schema 的天然缺陷。

在这个文档中，可以描述：

- 某个业务域的核心流程是什么
- 涉及哪些系统、哪些库、哪些表
- 表之间如何关联
- 每个字段在业务上的真实含义
- 统计口径是什么
- 哪些字段可以直接用于查询，哪些字段不能直接用
- 常见问题应该如何取数

这相当于在数据库上再加一层“业务知识层”。

## 3.2 给字段建立 tag 和可信度体系

这是一个非常有价值的方向。因为企业里很多字段的含义并不清晰，不能简单依赖注释。

你提出的做法可以扩展为：

- 给每个字段打业务标签，如“用户标识”“订单金额”“支付状态”“时间字段”
- 给字段打可信度等级，如“低可信”“中可信”“高可信”
- 标记字段说明来源，如“schema 注释”“skills.md”“人工确认”“历史查询反馈”
- 基于使用效果持续更新可信度

这会让系统不是一次性建模，而是一个不断演进的知识系统。

## 3.3 从问答系统升级为数据治理入口

如果继续往下做，这个平台不仅是“查数工具”，还会自然演化为：

- 企业数据资产目录入口
- 业务语义层入口
- 数据血缘查看入口
- 字段治理和口径治理入口
- AI 辅助的数据消费入口

也就是说，NL2SQL 只是入口，真正的长期价值是企业级数据智能中台。

## 4. 建议的整体架构

建议把系统拆成 6 层。

## 4.1 数据连接层

负责连接多个异构数据源，例如：

- MySQL
- PostgreSQL
- Oracle
- SQL Server
- Hive / Doris / ClickHouse
- API 数据源或中间数据服务

这一层负责：

- 数据源注册
- 连接管理
- 权限控制
- schema 拉取
- 元数据同步

## 4.2 元数据与语义层

这一层是系统基础，负责统一管理：

- 数据库
- 表
- 字段
- 字段类型
- 注释
- 主外键关系
- 表间关联
- 业务域归属
- 指标定义
- 维度定义

同时加入语义增强信息：

- 字段别名
- 业务含义
- 常用筛选值
- 同义词
- 禁用字段
- 推荐关联路径
- 默认时间口径

## 4.3 知识层

这就是你说的 `skills.md` 所在的位置。建议不要只是一份松散文档，而是逐步演进为“结构化业务知识库”。

知识来源可以包括：

- `skills.md` 业务文档
- 数据库字段注释
- 历史 SQL
- 数据分析师沉淀的口径文档
- 人工标注
- 用户反馈

知识内容建议覆盖：

- 业务流程说明
- 表用途说明
- 字段业务解释
- 指标定义
- 跨表 join 规则
- 常见分析场景模板
- 风险字段说明

## 4.4 查询规划层

这一层是 NL2SQL 的大脑，负责把用户问题转为可执行的查询计划。

主要步骤可以是：

1. 理解用户意图
2. 识别业务域
3. 识别关键实体、指标、维度、筛选条件、时间范围
4. 在语义层中召回候选表和字段（跨库召回）
5. 结合 `skills.md`、字段 tag、可信度进行重排序
6. 选择 join 路径或跨库组合方案
7. 生成 SQL 或多段子查询计划（针对跨库场景，可能生成分布式查询计划）
8. 执行前做安全校验
9. 返回数据、解释过程和图表建议

如果一个问题不能直接生成单条 SQL，还可以拆成：

- **多库分别查询**：大模型规划多个子查询分别去对应的物理库执行。
- **中间结果汇总**：利用 Presto/Trino 等联邦查询引擎，或在应用层内存中进行二次 Join 和聚合。
- **最终结果组装**：结合业务逻辑将跨库数据拼接为最终展示的图表或表格。

这对于跨库场景非常重要，也是本终极架构设计的核心壁垒。

## 4.5 治理与反馈层

这一层决定系统能不能越用越准。

建议设计如下能力：

- 字段可信度管理
- tag 维护
- 用户反馈纠错
- SQL 正确率评估
- 字段命中率统计
- 错误查询回溯
- 热门问题沉淀
- 低质量字段识别

字段可信度可以参考以下来源：

- schema 原始注释
- `skills.md` 描述
- 人工确认
- 高频成功查询
- 多次用户正向反馈
- 数据分析师审核通过

## 4.6 展示与交互层

面向员工的使用界面可以提供：

- 类似聊天框的自然语言提问
- 结构化条件补全
- SQL 解释和可视化说明
- 查询结果表格
- 自动推荐图表
- 查询来源说明
- 字段可信度提示
- 结果可导出

如果系统不确定，也应该明确告诉用户：

- 当前答案引用了哪些表
- 哪些字段解释可信度较低
- 是否存在多个可能口径
- 是否建议人工确认

## 5. 为什么 `skills.md` 是关键设计

你的思路里，`skills.md` 其实相当于 AI 的“业务外挂知识”。

单靠 schema 通常只能知道：

- 表名
- 字段名
- 数据类型
- 部分注释

但 AI 真正要准确查数，还需要知道：

- 这个表是不是核心事实表
- 这个金额字段是下单金额、支付金额还是退款后净额
- 这个时间字段是创建时间、支付时间还是完成时间
- 两张表应该如何 join
- 什么字段虽然像状态字段，但实际上已经废弃
- 业务里“新增用户”“有效订单”“活跃商家”分别怎么定义

所以建议 `skills.md` 采用模板化管理，例如：

```md
# 业务域：订单分析

## 业务说明
- 订单主流程：下单 -> 支付 -> 发货 -> 完成 -> 售后

## 核心表
- order_info：订单主表
- order_item：订单明细表
- payment_record：支付流水表

## 关键字段解释
- order_info.order_id：订单唯一标识，高可信
- order_info.user_id：下单用户 ID，高可信
- order_info.pay_amount：实际支付金额，高可信
- order_info.status：订单状态，需结合状态码表解释

## 关联关系
- order_info.order_id = order_item.order_id
- order_info.order_id = payment_record.order_id

## 指标口径
- 支付订单数：支付成功订单去重计数
- GMV：支付成功订单的支付金额总和，不含退款

## 常见问题
- 某时间段支付订单数
- 某产品线 GMV
- 某区域退款率
```

从长期来看，建议把 `skills.md` 从文档逐步升级为结构化配置，比如：

- YAML / JSON 知识配置
- 后台可维护的知识管理系统
- 向量检索 + 结构化规则混合召回

## 6. 字段 tag 与可信度体系建议

这部分是项目能否真正落地的关键设计之一。

每个字段建议维护以下信息：

- 基础信息：库、表、字段名、类型、注释
- 业务标签：ID、金额、时间、状态、枚举、类目、组织、地域等
- 业务归属：订单域、用户域、库存域、财务域等
- 含义描述：字段真实业务解释
- 示例值：帮助 AI 判断字段用途
- 可用性：可查询、敏感、废弃、仅内部使用
- 可信度：低 / 中 / 高
- 来源：schema、skills、人工确认、历史反馈
- 更新时间

可信度升级策略可以这样设计：

- 初始只有 schema 推断时，记为低可信
- 被 `skills.md` 明确说明后，升级到中可信
- 被数据分析师或业务方确认后，升级到高可信
- 被多次成功查询验证后，进一步提高权重
- 如果用户频繁反馈错误，则降低可信度

这样系统就具备“自学习”的基础。

## 7. 数据治理与数据血缘为什么必须做

你提到数据治理和数据血缘，这个判断是对的。

因为只做 NL2SQL，系统可能会“能查”，但不一定“查得对”。只有引入治理和血缘，系统才能在企业里真正可用。

数据治理主要解决：

- 字段命名混乱
- 口径不统一
- 重复指标太多
- 废弃表字段仍被误用
- 敏感数据缺乏控制

数据血缘主要解决：

- 这个指标来自哪张源表
- 中间是否经过清洗、聚合、派生
- 两个指标口径为什么不同
- 某个字段为什么能代表这个业务概念
- 查询结果是否依赖不稳定链路

建议后续为平台增加：

- 表级血缘
- 字段级血缘
- 指标级血缘
- SQL 来源血缘
- 报表反向追溯

- 字段 tag 管理

## 9. 一个可执行的查询流程示例

用户问题：

> 最近 30 天华东地区支付成功订单的 GMV 和订单数，按产品线分组展示柱状图。

系统处理流程：

1. 识别指标：GMV、订单数
2. 识别时间范围：最近 30 天
3. 识别筛选条件：华东地区、支付成功
4. 识别分组维度：产品线
5. 在知识层召回“订单域”相关表
6. 根据可信度优先选取高可信字段，如支付时间、支付金额、地区、产品线
7. 根据 `skills.md` 选择正确的口径定义
8. 生成 SQL 或跨库查询计划
9. 执行并生成结果表格
10. 自动推荐柱状图
11. 展示解释信息：使用了哪些表、哪些字段、口径是什么

这个“解释信息”非常重要，它决定用户是否信任结果。

## 10. 项目落地时最大的风险

建议你一开始就意识到以下风险：

## 10.1 不是 SQL 生成难，而是业务理解难

大模型写 SQL 本身不是最大问题，最大问题是：

- 选错表
- 用错字段
- 用错时间口径
- join 路径错误
- 聚合口径错误

## 10.2 跨库查询会明显增加复杂度

跨库会带来：

- 不同数据库方言差异
- 网络性能问题
- 权限隔离问题
- join 无法直接下推
- 结果一致性问题

所以一开始建议优先支持：

- 单业务域
- 少量核心库
- 高价值分析场景

## 10.3 企业场景一定要做权限控制

并不是所有员工都能访问所有数据，所以必须设计：

- 行级权限
- 列级权限
- 敏感字段脱敏
- 数据源访问授权
- 查询审计日志

## 10.4 可解释性决定用户是否敢用

如果系统只返回结果，不说明依据，用户不敢信。

因此必须能解释：

- 为什么选了这些表
- 为什么选了这些字段
- 哪些字段可信度低
- 最终 SQL 是什么
- 口径依据来自哪里

## 11. 系统核心功能点清单（多库终极版）

为了实现这个宏大的架构愿景，我们将系统拆解为以下几个主要的产品模块和功能点。这是构建企业级数据中台的标准组件：

### 11.1 前端：全局数据问答端 (C端)
- **自然语言全局搜索框**：像 Google 一样的单框搜索，不限制数据库，直接提问。
- **意图确认面板**：如果系统识别到可能跨库或涉及歧义口径，在执行前会弹窗让用户选择（例如：“您指的是 MySQL订单库的`订单数` 还是 Oracle财务库的`入账订单数`？”）。
- **查询路径（血缘）可视化**：在结果上方，用树状图或流程图的形式展示“本次数据是如何从 A 库和 B 库抽出来、经过哪些中间步骤拼接在一起的”。
- **多模态结果与数据大屏看板**：不仅支持单次表格和图表展示，还支持用户将高频查询的图表“一键添加到个人看板”，形成持续更新的数据大屏。
- **反馈与纠错面板**：不仅可以给结果点赞踩，还可以针对具体的某个字段解释进行纠正（“这个金额不含税，你们说错了”）。

### 11.2 后端：全局治理与调度引擎 (B端核心)
- **多源异构数据集成平台**：
  - 支持直连多种类型的关系型和非关系型数据库（MySQL, Oracle, ClickHouse 等）。
  - 全局元数据统一抓取与定时刷新。
- **联邦查询执行引擎**：
  - 内置或对接 Presto/Trino，负责解析并执行跨库的分布式 SQL。
  - 分布式查询状态监控、慢查询熔断与降级策略。
- **结构化业务知识网络（进化版 `skills.md`）**：
  - 可视化的知识图谱编辑器：分析师可以在界面上拖拽表名和字段，连线定义跨库的 Join 关系。
  - 全局口径字典：统一管理企业的核心指标口径（如“什么是 GMV”、“什么是活跃用户”），避免各库口径打架。
- **全局字段血缘与 Tag 治理工作台**：
  - 跨库同名同义/同名异义检测：自动扫描各库，发现潜在的语义冲突并报警。
  - 自动化血缘分析图谱：点击一个指标，能溯源到它最初是从哪个库的哪几张表计算得出的。
  - 基于 AI 预测的可信度评分雷达图：不再只依靠人工调整，AI 自动根据查询频率和报错率动态计算评分。
- **企业级权限路由网关**：
  - 对接企业内部的 SSO（单点登录）和 LDAP 组织架构。
  - 基于角色（RBAC）和基于属性（ABAC）的细粒度数据访问权限（列级脱敏、行级过滤）。

## 12. MVP 建议

如果你现在要开始做，建议不要一开始就做“大而全”，而是做一个能验证价值的 MVP。

推荐 MVP 范围：

- 先接 1 到 2 个核心数据库
- 先选 1 个业务域，例如订单域或用户域
- 先做 20 到 50 个高频问题
- 先沉淀第一版 `skills.md`
- 先建立字段 tag 和可信度模型
- 先支持结果表格 + 简单图表
- 先做人工确认和反馈闭环

MVP 的目标不是“什么都能问”，而是：

> 在一个业务域内，把高频数据问题查准、讲清、可验证。

## 13. 一句话总结
