# 企业级 NL2SQL 平台项目构想（单库多表 MVP 版）

## 1. 项目定位与背景

基于我们对“多数据库、多表联合查询”终极愿景的讨论，我们意识到跨库联邦查询（涉及异构数据源和极高的语义冲突）在初期落地时风险极高。因此，本篇文档聚焦于项目的 **MVP 落地阶段：多库接入，单库多表查询**。

在这一阶段，系统的使用模式是：

1. 系统后台配置了多个数据源（例如 MySQL 订单库、PostgreSQL 用户库、Oracle 财务库）。
2. 员工在使用时，**必须通过界面下拉框明确选择“当前要查询的数据库/业务域”**。
3. 用户输入大白话，大模型仅在**当前选定的单库范围内**，进行多表联合查询，生成 SQL，并返回数据或图表。

这种模式极大地降低了大模型的认知负担，同时完全保留了项目最核心的创新点（`skills.md`、字段 tag、可信度体系、数据治理）。

## 2. 核心创新点在“单库多表”下的应用

### 2.1 动态检索 Schema RAG (解决单库表数量过多的问题)

在一个企业级单库中，往往存在几百上千张表。如果把所有表结构硬塞给大模型，由于上下文窗口限制以及注意力稀释，模型很容易产生幻觉。我们采用 **Schema RAG (检索增强生成)** 架构：

- **向量化**：系统将选定库中所有表的表名、字段、注释以及 `skills.md` 描述向量化，存入向量数据库。
- **动态召回**：当用户提问时，系统先进行相似度检索，**精准召回与当前问题最相关的 Top 5\~10 张表**。
- **专注生成**：只把这 5\~10 张表作为 Context 喂给大模型生成单库 SQL，既支持了庞大的数据库，又保证了 SQL 生成的高准确率。

### 2.2 引入业务外挂知识：`skills.md`

单靠 schema 无法应对复杂的业务提问。在单库场景下，我们依然需要为该库配置 `skills.md`（业务语义说明书）。

在这个文档中记录：

- 该库的核心主流程
- 事实表与维度表的明确区分与关联关系（Join 规则）
- 关键字段的真实业务含义和统计口径
- 哪些字段可用，哪些是废弃/冗余字段

### 2.3 字段 Tag 与可信度自学习体系

由于很多字段没有注释或注释不准，系统需要给每个字段建立动态档案：

- **Tag 打标**：标记字段是“金额类”“时间类”“枚举状态类”等。
- **可信度流转**：
  - 仅有 Schema 注释：**低可信**
  - 在 `skills.md` 中被定义：**中可信**
  - 被数据分析师打 Tag 确认，或在用户历史查询中被多次成功验证：**高可信**
  - 发生报错或用户点击“结果不准”：**可信度降级**

这让系统不再是静态的查数工具，而是具备“人机协同学习”能力的活系统。

### 2.4 轻量级数据治理与血缘闭环（单库依然必不可少）

即便在单库范围内，表结构的混乱（如同名字段不同口径、废弃字段遗留）也会导致大模型频频翻车。因此，必须将数据治理与血缘追踪融入系统的核心流程中：

- **基于血缘的查询自愈**：通过大模型生成的 SQL 记录，自动逆向解析并沉淀出“单库表级/字段级血缘”。当数据库某张表的字段发生类型变更或被删除时，系统通过血缘图谱立刻告警，并提示修改对应的 `skills.md`，防止下一次问答报错。
- **治理探针与废弃隔离**：在 Tag 体系中设立“废弃/敏感/不推荐”标签。对于历史遗留的脏表和脏字段，通过治理工作台将其隔离，在 RAG 召回时直接屏蔽，避免污染大模型的上下文。

## 3. 建议的整体架构（单库多表版）

### 3.1 数据接入与预处理层

- 注册多个异构数据源，提取单库的 Schema。
- 将单库 Schema 与对应的 `skills.md` 解析后，写入向量数据库（如 Milvus / Chroma）。

### 3.2 语义理解与召回层 (Schema RAG)

- 识别用户意图中的时间、维度、指标、筛选条件。
- 在当前选定的**单一数据库**内，召回相关的表和字段（Top 5-10）。

### 3.3 AI Agent 层（基于 Python / LangGraph 编排）

抛弃传统的 Java 线性调用，我们引入 **LangGraph** 构建带状态（Stateful）和自我纠错循环（Cyclic）的智能体工作流。
整个 NL2SQL 过程被定义为一个有向图（Graph），包含：

- **状态 (State)**：维护用户的原始问题、当前选中的数据库、召回的 Schema、生成的 SQL、执行报错信息以及重试次数。
- **节点 (Nodes)**：
  1. `Schema_Retriever_Node`：去向量库召回相关的表结构和 `skills.md`。
  2. `SQL_Generator_Node`：大模型根据 Schema 写出 SQL。
  3. `SQL_Executor_Node`：在沙箱中执行 SQL。如果报错，提取 Error Message。
  4. `Data_Visualizer_Node`：大模型根据执行成功的数据，决定画什么图（Echarts Option）。
- **条件边 (Conditional Edges)**：
  - `SQL_Executor_Node` 执行后，如果**成功** -> 流转到 `Data_Visualizer_Node`。
  - 如果**失败且重试次数 < 3** -> 携带 Error Message 流转回 `SQL_Generator_Node` 让大模型自我修复（Self-Correction）。
  - 如果**失败且重试次数 >= 3** -> 抛出异常给用户。

### 3.4 交互与反馈层 (闭环)

- 展示数据表格或自动推荐 Echarts 图表。
- **可解释性展示**：明确告诉用户“本次查询使用了 `order_info` 和 `user` 表，依赖了高可信字段 `pay_amount`”。
- **反馈收集**：用户点赞/踩，驱动底层字段可信度的升降级。

## 4. `skills.md` 模板示例（针对单库）

```md
# 数据库：trade_db (交易域)

## 核心表说明
- order_info：订单主表（事实表），包含所有主订单信息。
- order_item：订单明细表，商品粒度的金额和折扣在这里。
- refund_record：退款记录表。

## 关键关联规则 (Join Path)
- 订单与明细：`order_info.order_id = order_item.order_id`
- 订单与退款：`order_info.order_id = refund_record.order_id`

## 关键口径与字段防坑
- **GMV**：取 `order_info.total_amount`，且 `status IN (2,3,4)`（支付成功及之后的阶段）。千万不要用 `create_amount`。
- **退款金额**：取 `refund_record.actual_refund`，且 `refund_status = 'SUCCESS'`。
- **废弃字段**：`order_info.old_user_id` 已废弃，一律使用 `user_id`。
```

## 5. 汇报专用：逻辑可行性推演与核心技术难点

在单库多表的 MVP 阶段，虽然避开了分布式查询的深水区，但要在企业内部真正用起来，依然面临以下逻辑漏洞和技术挑战。这是我们在项目答辩/汇报时必须重点阐述的防坑设计：

### 5.1 Schema RAG 的“一子错，满盘皆输”问题

- **逻辑漏洞**：用户问“华东区退款率”，如果 RAG 阶段没有召回 `refund_record`（退款表），而是召回了 `order_info`（订单表），大模型生成的 SQL 必然是错的（比如用订单状态去算退款率）。在 RAG 链路中，**召回错了，生成一定错**。
- **技术难点**：单纯依赖向量检索（如查询语句和表注释的 Cosine 相似度）是不够的。必须引入 **Hybrid Search（混合检索）**，结合“关键词 BM25 + 向量检索 + 历史高频 SQL 模板匹配”，甚至让大模型在生成前先进行一步“思考（Chain of Thought）”，自己决定需要哪些表。

### 5.2 可信度体系的“冷启动”与“恶向反馈”风险

- **逻辑漏洞**：项目初期，所有字段都是低可信度，AI 容易乱用字段导致体验差，用户不用，就更没有反馈，陷入死循环。同时，如果某个用户出于误解，不断给正确的数据点“踩”，会导致高可信字段被错误降级。
- **技术难点**：
  1. **冷启动突破**：不能干等用户反馈，必须在系统上线前，由数据分析师结合现有 BI 报表的底层 SQL，批量解析出高频关联字段，直接将其预置为“高可信”。
  2. **防恶意/误操作算法**：用户的“踩”不能直接扣减字段分数，而是引入“反馈权重池”，普通用户的踩只触发“审核队列”，只有管理员/分析师确认后，或者超过一定阈值的群体反馈，才能真实降级字段可信度。

### 5.3 极其复杂的业务 SQL（嵌套与聚合）生成稳定性

- **逻辑漏洞**：大白话往往隐藏着极度复杂的嵌套逻辑。例如：“把昨天每个部门业绩排名前三的员工拉出来”。这在 SQL 中需要用到 `ROW_NUMBER() OVER(PARTITION BY)` 窗口函数。如果只是把 Schema 给大模型，它经常会用简单的 `GROUP BY` 和 `LIMIT`，导致逻辑完全错误。
- **技术难点**：大模型对复杂窗口函数、多重子查询的 Zero-shot（零样本）生成能力依然不稳定。因此，`skills.md` 中不能只有表结构和口径，必须包含 **Few-shot（少样本）的高级 SQL 模板**。当意图识别到“排名前几”、“连续三天”等字眼时，动态注入这些模板指导大模型写出复杂 SQL。

### 5.4 动态 SQL 注入与数据越界查询风险

- **逻辑漏洞**：员工可能会说“删除订单表里今天的记录”或者“给我看总裁办的工资明细”。大模型如果真的生成了 `DELETE` 或绕过权限的查询，后果不堪设想。
- **技术难点**：必须在 AI 生成 SQL 后、数据库执行前，加入一个坚固的 **SQL 拦截网关（SQL Parser & Validator）**：
  1. 基于 AST（抽象语法树）解析，拦截所有非 `SELECT` 类型的语句。
  2. 强行注入当前用户的数据隔离范围（比如在 WHERE 后面强制拼装 `AND department_id = '本部门'`）。

## 6. 系统核心功能点清单（MVP 落地版）

为了将上述构想落地，系统需要拆解为以下具体的前后端功能模块：

### 6.1 前端：员工问答端 (C端)

- **数据源切换器**：顶部下拉框，用于选择当前要查询的业务域/数据库（如“订单域”、“财务域”）。
- **自然语言对话框**：支持文本输入、语音输入（可选）、联想词提示、历史提问记录快速点击。
- **查询过程白盒展示**：在出结果前，流式展示 AI 的思考过程（例如：“正在召回订单表...”、“已生成查询 SQL...”），提升用户信任感。
- **多模态结果展示**：
  - 数据表格（支持分页、排序、导出 Excel/CSV）。
  - 自动推荐的 Echarts 图表（柱状图、折线图、饼图等，支持一键切换图表类型）。
- **可解释性与溯源面板**：结果旁附带“口径说明”，展示本次查询用到了哪些表、哪些高可信字段，依据了哪条 `skills.md` 规则。
- **用户反馈机制**：每条结果下方提供“赞/踩”按钮，并支持点击“踩”后选择原因（如：数据不准、图表类型错、口径不对）。

### 6.2 后台：治理与管理端 (B端)

- **数据源与 Schema 管理**：
  - 数据库连接配置（JDBC URL、账号密码）。
  - Schema 自动一键拉取与定时同步。
  - 表与字段的启用/禁用控制（屏蔽系统表和敏感表）。
- **`skills.md`** **业务知识库管理**：
  - 在线 Markdown 编辑器，支持按业务域编写表关系、业务口径、核心指标定义。
  - 知识库的向量化（Embedding）触发按钮与状态查看。
- **字段 Tag 与可信度工作台**：
  - 字段打标面板：为字段批量添加业务标签（如“金额”、“时间”、“用户ID”）。
  - 可信度手动调整：数据分析师可直接将某字段设为“高可信”。
  - **用户反馈处理中心**：集中查看 C 端用户的“踩”，分析师确认后决定是否对某字段进行降级。
- **查询审计与 SQL 实验室**：
  - 查询日志列表：记录谁、在什么时间、问了什么问题、AI 生成了什么 SQL、是否报错。
  - 慢查询分析：针对执行超过 5 秒的 SQL 进行记录与预警。
- **安全与权限控制**：
  - 用户角色管理（普通员工、分析师、管理员）。
  - 库级别的数据访问授权。

### 6.3 后台：单库数据治理与血缘引擎 (治理端核心)

在单库几百张表的环境下，若不进行治理，NL2SQL 系统将成为“垃圾进、垃圾出”的重灾区。因此必须提供专门的治理与血缘模块：

- **自动化单库血缘图谱**：
  - **动态血缘解析**：系统后台定时解析大模型生成的成功 SQL（以及历史高频查询日志），自动逆向生成一张表与表、字段与字段之间的有向无环图（DAG），展示“经常被联合查询的表簇”。
  - **变更级联预警**：当 DBA 修改了某张表的结构（如删除了某个旧状态字段），血缘引擎会立即报警，提示管理员“该字段被高频查询，且在 `skills.md` 中被引用，可能导致后续 AI 生成报错”。
- **冗余字段与废弃表隔离（库内治理）**：
  - **废弃字段雷达**：扫描库中常年未被查询命中、或者被分析师打上“弃用” Tag 的字段，生成待清理报表。
  - **白名单/黑名单机制**：直接将某些“测试表”、“历史归档表”放入召回黑名单，彻底避免大模型检索到这些表而产生幻觉。
- **单库口径一致性校验看板**：
  - 如果一个业务指标（如 GMV）既能从 `order_info` 算出来，也能从 `payment_record` 算出来，系统会提示存在“口径分歧风险”，并建议在 `skills.md` 中强制指定唯一权威口径。

## 7. 完整的项目开发路径（Roadmap）

为了确保项目稳步落地，建议将整个开发周期分为以下四个阶段。每一步都明确前后端的开发任务和交付物：

### 阶段一：核心链路贯通（MVP 验证期）

**目标**：不搞复杂的系统管理，先证明“大模型 + skills.md”能在一个单库中稳定生成复杂 SQL 并查出正确数据。

- **后端架构重构（微服务分离）**：
  - **Python (LangGraph/FastAPI)** 负责专门的 AI 编排，定义 State、Nodes (生成 SQL, 执行 SQL) 和自我修复循环。
  - **Java (Spring Boot)** 负责基础业务逻辑，比如权限控制、前端接口代理。
- **核心任务**：
  - 在 Python 端用 LangGraph 编写一个极简的有向图：`生成SQL -> 校验并执行 -> 如果报错带着错误信息回去重写`。
  - 硬编码读取本地的 `skills.md` 文件和数据库 Schema。
- **前端任务**：
  - 开发极简的“对话问答框”和“数据表格展示”页面。
- **阶段产出**：一个基于 LangGraph 的能通过自然语言查出单库数据并能**自我纠错**的 AI 服务。

### 阶段二：引入 Schema RAG 与动态配置（进阶拓展期）

**目标**：解决单库表太多导致大模型幻觉的问题，引入向量数据库，并将静态配置转为动态系统管理。

- **后端任务**：
  - 接入向量数据库（如 Milvus / Chroma）。
  - 开发 Schema 自动拉取与向量化脚本（将几百张表的注释、`skills.md` 存入向量库）。
  - **核心逻辑**：实现“自然语言提问 -> 向量检索召回 Top 5 相关表 -> 拼装 Prompt -> 生成 SQL”的 RAG 完整链路。
  - 开发后台管理接口：支持在页面上配置数据库连接、在线编辑 `skills.md`。
- **前端任务**：
  - 开发后台管理页面（数据源管理、知识库在线编辑）。
  - 前台增加“查询过程白盒展示”（让用户看到正在召回哪几张表）。
- **阶段产出**：能够支撑单库 200+ 张表检索的 NL2SQL 系统，且具备后台配置能力。

### 阶段三：可信度闭环与图表可视化（用户体验期）

**目标**：解决“AI 胡说八道”和“展示干瘪”的问题，建立字段 Tag、可信度体系以及丰富的图表。

- **后端任务**：
  - 设计并创建数据表，用于存储“字段 Tag”、“字段可信度评分”和“用户反馈记录”。
  - 修改 Prompt 逻辑，在生成 SQL 时强行注入“优先使用高可信字段”的规则。
  - 增加大模型生成 Echarts 图表配置（Option）的接口。
- **前端任务**：
  - 对话框结果区增加自动生成的 Echarts 图表展示。
  - 结果区下方增加“赞/踩”反馈按钮及原因弹窗。
  - 开发后台“字段 Tag 打标工作台”和“用户反馈审核中心”页面。
- **阶段产出**：带有图表展示、可人工纠错、可信度动态升降的完整闭环问答系统。

### 阶段四：单库治理与血缘沉淀（企业级深化期）

**目标**：解决脏数据表污染和长效运维问题，构建单库血缘与隔离机制，这也是本项目的核心护城河。

- **后端任务**：
  - 开发定时任务：解析历史成功的 SQL 记录（AST 抽象语法树解析），自动构建单库内表与字段的有向无环图（血缘 DAG）。
  - 开发黑白名单拦截网关：在 RAG 召回前，强制过滤掉被打上“废弃”标签的脏表。
  - 增加非 SELECT 语句（如 DROP/DELETE）的安全拦截拦截器。
- **前端任务**：
  - 开发后台“单库血缘关系可视化图谱”页面（使用 Echarts 关系图或 G6 等图库）。
  - 开发“废弃字段/隔离表”的管理列表。
- **阶段产出**：具备自动血缘追踪、废弃表隔离和高安全性拦截的企业级智能查数平台。

***

> **总结**：单库多表版完全保留了你的核心思想，但通过“锁定上下文边界”大幅提高了 AI 生成的准确率和项目的可落地性。这是企业级 NL2SQL 走向成功的必经阶段。
